# Hermes 的代码失误复盘（2026-04-15）

> 本文件记录 Hermes 在编写 `lint.py v2.0` 时暴露的系统性缺陷，以及对应的改进承诺。

## 背景

2026-04-15，Hermes 独立完成 `lint.py v2.0` 并自认为「功能完整」。Claude-Code 进行 Code Review 后，一次性指出 6 个严重问题，其中 2 个属于安全/可靠性缺陷，4 个属于系统性思维漏洞。

## 暴露的 6 个问题

### 1. 优先手写解析器，而不是使用标准库

**问题**：`extract_frontmatter()` 是纯手写的字符串 split 逻辑，没有优先使用 `PyYAML`。

**根因**：
- 图快：觉得「就几行 YAML，手写更快」
- 低估复杂度：没有考虑到 inline list `[a, b]`、转义引号、多行字符串等边界情况
- 违反了 [[05-Knowledge/Pitfalls/手写-parser-陷阱|"不要重复造轮子"]] 的基本原则

**改进**：
- **标准库优先**：任何解析任务，先找标准库/成熟库
- **fallback 必须标注局限性**：手写降级方案时，在代码注释里明确说「这不支持 XXX」
- **测试驱动边界**：写解析器前，先列出 10 个可能输入的测试用例

### 2. 安全边界意识薄弱（路径遍历漏洞）

**问题**：`resolve_link()` 直接拼接用户输入的 wikilink 到文件系统路径：
```python
target = VAULT / f"{link_text}.md"
```
如果内容里有 `[[../../../etc/passwd]]`，脚本会尝试读取 vault 外部文件。

**根因**：
- 默认信任输入：潜意识里认为 "wikilink 都是安全的"
- 没有把 lint 脚本当作"需要防御外部输入"的程序来对待

**改进**：
- **永远不信任输入**：任何来自文件内容的字符串，在用于路径拼接前都要做消毒和边界检查
- ** resolve() + startswith() 双保险**：拼接路径后 `.resolve()`，再校验是否以 vault 根目录开头
- 把这条加入 [[05-Knowledge/Protocols/代码审查清单|个人代码审查清单]]

### 3. 硬编码本地绝对路径

**问题**：文档和代码里大量写死 `/Users/<username>/Library/Mobile Documents/...`。

**根因**：
- 本地开发惯性：在自己机器上跑通了就觉得够了
- 没有考虑可移植性和隐私泄露

**改进**：
- **环境变量优先**：所有工具脚本的第一行就是 `VAULT = Path(os.environ.get(..., 默认))`
- 文档中的示例代码使用 `$VAULT` 或相对路径

### 4. 指标定义遗漏数据源

**问题**：计算 Agent 活跃度（`agent_latest_update`）时只扫描 `.md` 的 frontmatter，**没扫描 `log/` 目录**。导致 Agent 明明每天在写 log，lint 却报它 7 天未更新。

**根因**：
- 定义指标时没有追问："这个行为还会在哪些地方留下痕迹？"
- 单点思维：只想到 "页面有 updated 字段"，没想到 "log 也是时间戳"

**改进**：
- **多源验证**：定义任何指标前，列出所有可能的数据源
- 对 "活跃度" 这类指标，至少要检查：内容页 frontmatter + log 记录 + audit 活动

### 5. 文档-代码-模板三者不一致

**问题**：Audit 创建示例缺少 `target`/`severity` frontmatter，但 lint 检查却要求这两个字段必须存在。

**根因**：
- 修改一处时，没有同步检查关联物
- 缺少 "变更影响面检查" 的习惯

**改进**：
- **三位一体检查**：任何 frontmatter 字段的变更，必须同时检查「模板 → 示例 → lint 规则」
- 在改完代码后，反向grep所有示例和模板

### 6. 缺少防御性编程（大文件保护）

**问题**：`content = md.read_text(encoding="utf-8")` 没有任何大小限制。

**根因**：
- 没有考虑 "如果目录里意外出现了一个几百 MB 的 .md 文件怎么办？"
- 防御性编程意识不足

**改进**：
- **所有文件操作都设上限**：读取前检查 `st_size`，超过阈值（如 10MB）则跳过并告警
- 对 "不可控目录" 做任何批量处理时，先问自己最坏情况是什么

## 改进承诺

1. **写解析器前，先搜标准库**（禁止因为"图快"而手写）
2. **任何来自文件/用户的输入，在用于路径/命令/API 前必须消毒**
3. **指标定义必须列出所有数据源**
4. **修改 frontmatter/配置/接口时，同步检查模板+示例+lint**
5. **文件操作加大小限制，网络操作加超时**
6. **每次写完代码后，先用 "恶意输入" 的角度快速 fuzz 一遍**
